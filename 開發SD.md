# FitSocial 系統設計文件 (SD)

## 1. 專案概述

FitSocial 是一個以「結構化運動紀錄」為核心的社群平台。旨在解決傳統運動 App 紀錄過於繁瑣的問題，並透過「課表學習」與「目標諮詢」建立高互動性的運動社群。

## 2. 核心功能需求 (Functional Requirements)

- **結構化紀錄系統**：
  - **有氧 (Aerobic)**：支援 LSD、間歇、配速跑；欄位包含組數、距離、時間。
  - **肌力 (Strength)**：按部位分組（胸、背、腿）；支援多動作、重量、次數紀錄。
- **低摩擦輸入 (Low-Friction UI)**：以標籤選擇 (Chips) 與滑桿取代大量文字輸入。
- **課表學習機制**：使用者可一鍵將他人的運動紀錄存為自己的訓練範本（學習課表）。
- **諮詢發問區 (Q&A)**：根據用戶目標（如：減脂、增加爆發力）自動媒合並推薦熱門課表。
- **數據可視化**：個人訓練負荷、連勝天數 (Streaks) 與成長趨勢圖。

## 3. 系統架構 (System Architecture)

建議採用前後端分離架構：

- **前端**：React.js / Next.js (搭配 Tailwind CSS 與 Lucide Icons)。
- **後端**：Node.js (Express) 或 Python (FastAPI)。
- **資料庫**：PostgreSQL (關聯式資料庫，適合處理結構化課表)。
- **快取**：Redis (處理動態牆排序與連勝計算)。

## 4. 資料模型設計 (Data Schema)

### 4.1 使用者 (Users)

| 欄位名     | 類型   | 描述         |
|------------|--------|--------------|
| id         | UUID   | 唯一識別碼   |
| username   | String | 用戶名       |
| following  | Array  | 關注列表     |
| streak_count | Int  | 連續運動天數 |

### 4.2 運動紀錄 (WorkoutLogs)

採用多型設計，區分有氧與肌力。

| 欄位名   | 類型   | 描述                                      |
|----------|--------|-------------------------------------------|
| id       | UUID   | 紀錄識別碼                                |
| user_id  | UUID   | 關聯使用者                                |
| type     | Enum   | aerobic 或 strength                       |
| category | String | 項目（如：LSD, 400m 間歇, 胸部）         |
| intensity| Enum   | 體感強度（Easy, Moderate, Hard）          |
| details  | JSONB  | 結構化數據（有氧：sets/dist；肌力：movements 陣列） |
| is_template | Boolean | 是否可供他人學習                          |

## 5. 核心邏輯流程 (Business Logic)

### 5.1 「學習課表」邏輯

- **觸發**：用戶點擊貼文上的「學習此課表」。
- **處理**：後端提取該 WorkoutLog 的 details 物件。
- **儲存**：在用戶的 SavedPrograms 資料表中建立一筆新關聯，複製對方的參數。
- **回饋**：通知原作者「有人學習了你的課表」，增加社交成就感。

### 5.2 「諮詢推薦」邏輯

- **分類索引**：系統預設標籤（如：有氧能力、肌力增長）。
- **查詢**：當用戶點選目標，系統檢索 is_template: true 且標籤匹配的熱門貼文。
- **排序**：依據「學習次數」或「按讚數」進行權重分配。

## 6. UI/UX 設計規範 (UI/UX Principles)

- **視覺語言**：大量使用 Rounded-2xl 以上的圓角，營造親和力。
- **操作優先**：三點完成：任何運動紀錄應在「選類型 -> 選強度 -> 填數值」三次主要點擊內完成。
- **無文字區**：心得紀錄設為選填，預設提供多組 Emoji 快選。
- **反饋機制**：運動完畢後的「數據摘要卡片」應具備高度的視覺美感，方便用戶截圖分享至 Instagram。

## 7. 後續深入開發方向 (Roadmap)

- **Phase 1**：核心紀錄功能與個人趨勢圖（已完成原型）。
- **Phase 2**：社群追蹤與課表複製功能。
- **Phase 3**：發問區自動化媒合與專家（教練）認證系統。
- **Phase 4**：穿戴式設備 (Apple Health / Garmin) 數據同步。

## Gemini 的小建議

在實作這份 SD 時，「資料庫的 JSONB 結構」 是關鍵。因為運動種類太多，如果每個動作都開一個欄位會太死板，用 JSONB 儲存肌力訓練的動作陣列，能讓你未來隨時新增「新的運動類型」而不需要改動資料庫架構。
